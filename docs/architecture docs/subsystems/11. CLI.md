# CLI

The Prisma Next CLI is the user-facing entry point into the framework's tooling plane. It wraps contract emission, database init/verify/sign, and on-disk migration management behind a single binary (`prisma-next`) and a programmatic control API for embedders. This doc describes the CLI as a subsystem: how it is distributed, what its command surface looks like, how its non-trivial flows (notably `init`) are structured, what it exposes programmatically, and where it sits in the package layering.

This doc is **architectural**. UX, copy, flag conventions, exit-code semantics, and JSON output rules live in the [CLI Style Guide](../../CLI%20Style%20Guide.md); the distribution shape lives in [ADR 211 — `prisma-next` bin-only distribution](../adrs/ADR%20211%20-%20prisma-next%20bin-only%20distribution.md). Cross-link, do not duplicate.

## Responsibilities

- Run developer-driven control-plane commands: emit contracts, initialise/verify/sign databases, plan and apply on-disk migrations, scaffold new projects.
- Provide a stable programmatic surface (`@prisma-next/cli/*`) so build tools, CI scripts, and extension authors can drive the same operations without spawning a subprocess.
- Bridge between author-time artefacts (`contract.json`, `contract.d.ts`, migration packages) and runtime targets via the framework's control API.

**Non-goals:** query execution (Runtime), capability lowering (Adapters), bundle policy for hosted preflight (Preflight & CI). The CLI invokes those subsystems; it does not own them.

## Distribution model

Two packages, one CLI:

| Package | Purpose | Public surface |
|---|---|---|
| [`prisma-next`](../../../packages/1-framework/3-tooling/prisma-next/package.json) | npm-distributed bin (`pnpm dlx prisma-next init`, devDep of generated apps) | `bin` only — no `exports`, no `main`, no `types`. `import 'prisma-next'` is a hard resolution failure. |
| [`@prisma-next/cli`](../../../packages/1-framework/3-tooling/cli/package.json) | Workspace CLI source-of-truth and programmatic-API target | `bin` plus `exports` for `.`, `./config-types`, `./control-api`, `./init-output`, and per-command modules under `./commands/*` |

`prisma-next/dist` is a verbatim copy of `@prisma-next/cli/dist` produced at build time, with bin entries `chmod +x`'d. Runtime deps are mirrored exactly. A drift-lint script enforces the invariants. See [ADR 211](../adrs/ADR%20211%20-%20prisma-next%20bin-only%20distribution.md) for the rationale (curated public README, future bundling option, asymmetric public/internal naming) and the [Package Naming Conventions § Framework Domain table](../../reference/Package%20Naming%20Conventions.md#path--package-name-examples) for where each lives in the workspace.

The asymmetry is deliberate: the **command** ships under the ergonomic unscoped name; **programmatic APIs** stay under the scoped name to signal that they participate in the internal stability contract.

## Command surface

Top-level commands are built from `commander.Command` instances in [`packages/1-framework/3-tooling/cli/src/cli.ts`](../../../packages/1-framework/3-tooling/cli/src/cli.ts). Subcommands are colocated under [`src/commands/`](../../../packages/1-framework/3-tooling/cli/src/commands/), one TypeScript module per subcommand, plus an `init/` directory for the larger `init` flow.

Current shape:

- `prisma-next init` — greenfield project scaffold (top-level; see [Init flow architecture](#init-flow-architecture))
- `prisma-next contract emit | infer` — produce `contract.json` + `contract.d.ts` from PSL or TS authoring
- `prisma-next db init | update | schema | verify | sign` — adopt, update, inspect, verify, and sign a target database against a contract
- `prisma-next migration new | plan | preflight | apply | show | status | ref` — author and apply on-disk migration packages

Each command file exports a `create<Name>Command()` factory that returns a `Command`. The root `cli.ts` registers parent groups (`contract`, `db`, `migration`) and adds the per-command subcommands. The same `create*Command` factories are also exposed from `@prisma-next/cli/commands/<name>` so embedders can compose a subset of the CLI into their own binary.

Style — flag naming, exit codes, JSON shape, help routing — is the [CLI Style Guide](../../CLI%20Style%20Guide.md)'s domain. This doc only describes *that* the surface exists and *where* it is implemented.

## Init flow architecture

`prisma-next init` is the most involved command and the only one with a non-trivial internal pipeline. It lives under [`src/commands/init/`](../../../packages/1-framework/3-tooling/cli/src/commands/init/) and is intentionally split into single-purpose modules so each concern is testable in isolation:

| Module | Concern |
|---|---|
| `inputs.ts` | Resolve target (Postgres / Mongo) and schema location from flags + interactive prompts; produce a fully-resolved input record |
| `detect-package-manager.ts` | Detect the project's package manager from lockfiles, `package.json#packageManager`, and falls back to npm |
| `detect-pnpm-catalog.ts` | Detect a pnpm workspace catalog so installed deps land in the catalog rather than as literal versions |
| `reinit-cleanup.ts` | Remove generated files when re-initialising over an existing config (`--force` required) |
| `templates/` | Static templates for `prisma-next.config.ts`, `prisma/contract.prisma`, `prisma/db.ts`, `.env.example`, `prisma-next.md`, `.agents/skills/prisma-next/SKILL.md` |
| `hygiene-*.ts` | Idempotent updates to `.gitignore`, `.gitattributes`, and `package.json` scripts |
| `probe-db.ts` | Optional connectivity probe before declaring success |
| `output.ts` | Arktype schema for the `--json` success document (re-exported from `@prisma-next/cli/init-output`) |
| `errors.ts`, `exit-codes.ts` | Init-specific PN error codes and command-specific exit codes |
| `init.ts`, `index.ts` | Orchestration and the `createInitCommand` factory |

The high-level pipeline is: resolve inputs → detect re-init / clean up under `--force` → write templates → update hygiene files → install deps via the detected package manager → emit the contract programmatically through the control API → optionally probe the database → render the success document. Each step is structured so a `--no-install` invocation can short-circuit dependency installation and contract emission while still scaffolding source files.

The user-facing details (which files land, what the prompts say, when `--force` is required, the celebratory after-init output) are owned by the [CLI Style Guide § Init Flow](../../CLI%20Style%20Guide.md#init-flow). This doc records the *internal decomposition* so contributors know where to make changes.

## Programmatic API

`@prisma-next/cli` exposes a small set of subpath entrypoints for embedders:

- **`@prisma-next/cli`** — top-level utilities (currently `createContractEmitCommand`, `loadContractFromTs`)
- **`@prisma-next/cli/config-types`** — `PrismaNextConfig`, `ContractConfig`, `defineConfig` (re-exported from `@prisma-next/config/config-types`)
- **`@prisma-next/cli/control-api`** — the programmatic control plane: `createControlClient`, standalone `executeContractEmit` / `executeDbInit` / `executeDbUpdate` / `executeDbVerify`, `enrichContract`, and the full set of option/result/failure types backing each operation
- **`@prisma-next/cli/init-output`** — the arktype schema for `init --json` success documents
- **`@prisma-next/cli/commands/<name>`** — individual `create<Name>Command()` factories for hosts that want to embed a subset of the CLI in their own binary

The programmatic surface mirrors the CLI command surface but is independently testable and does not require spawning a process. Build tools, CI scripts, and IDE integrations should prefer it over shelling out, both for performance and because it returns typed result/failure documents instead of parsing stdout.

The shim package (`prisma-next`) deliberately does **not** participate in this surface. ADR 211 records the rationale: a second public import surface here would force running stability management in parallel for the same APIs. Programmatic consumers must depend on `@prisma-next/cli` directly.

Versioning: `prisma-next` and `@prisma-next/cli` ship in lockstep — same version, mirrored deps, enforced by the drift lint. Programmatic consumers should pin both to the same version when transitively depending through a freshly-init'd app.

## Layering and plane

In the workspace's domain/layer/plane model (see [`architecture.config.json`](../../../architecture.config.json) and [Package Layering](../Package-Layering.md)):

- **Domain:** `framework` (target-agnostic)
- **Layer:** `tooling` (3-tooling)
- **Plane:** `migration`

This places the CLI above `core/` and `authoring/` (which it imports from) and below `runtime/` and `adapters/` (which it does **not** import directly — the control API mediates target-specific work). Concretely, `@prisma-next/cli` may depend on `@prisma-next/contract`, `@prisma-next/config`, `@prisma-next/emitter`, `@prisma-next/framework-components`, `@prisma-next/migration-tools`, `@prisma-next/psl-printer`, `@prisma-next/utils`, and `@prisma-next/errors`. It must not import target-family or runtime packages directly; target-specific behaviour reaches the CLI through the control API and target/adapter facades configured in `prisma-next.config.ts`.

The shim package (`prisma-next`) sits at the same layer and depends only on `@prisma-next/cli` (devDep, for the build-time copy) plus the same runtime deps mirrored verbatim. `pnpm lint:deps` enforces this; ADR 211's drift lint enforces the dep-mirroring invariant on top.
