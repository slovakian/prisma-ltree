### F-1 — 📝 Follow-up — QA script steps need workspace-resolution note

**Scenario:** Pre-flight / Scenarios 1–5 (substrate setup)
**Step:** Step 2 of each scenario
**Oracle:** Steps are copy-pasteable and run correctly as written in the script.
**Observed:**

```
Error [ERR_MODULE_NOT_FOUND]: Cannot find package '@prisma-next/sqlite' imported from /private/tmp/prisma-next-qa-2614/sqlite-hang-exit.mts
```

When scripts are written to `/tmp/...`, `node --import tsx/esm` cannot resolve `@prisma-next/*` workspace packages because pnpm links them relative to each workspace package's `node_modules/`, not the root. Scripts must be placed and run from within a workspace member directory (e.g., `examples/prisma-next-demo-sqlite/`) to resolve workspace imports.

**Expected (per script):** Scripts run from `/tmp/prisma-next-qa-2614/` with `NODE_PATH=$REPO/node_modules` resolving workspace packages.
**Reproduction:**
- `git rev-parse HEAD` → `b495cef6d47ba561d71354c580e479c8c31a057f`
- `git status` at failure → clean
- Exact command: `NODE_PATH="$REPO/node_modules" node --import tsx/esm /tmp/prisma-next-qa-2614/sqlite-hang-exit.mts`
**Notes:** This is a QA-script authoring issue, not an implementation bug. The fix is to update the script's "Steps" sections to direct the runner to create scripts inside an existing workspace member directory (e.g., `examples/prisma-next-demo-sqlite/`) instead of `/tmp`. The runner worked around this during the run; the underlying implementation behaved correctly in all cases.

## Exploratory notes

**Probes run:**

1. `close()` before `runtime()` is called (Postgres): No error, subsequent `runtime()` throws `"Postgres client is closed"`. Idempotent: second `close()` is also a no-op. ✅ Expected.

2. `close()` called immediately after `runtime()` without awaiting the microtask queue: The pool creation is initiated by `runtime()` but `close()` called synchronously (before the microtask queue flushes) still resolves cleanly and locks the facade. ✅ Expected — the implementation handles this via `closed = true` set first, then `await connectPromise?.catch(...)`.

3. Attempted Mongo exploratory probing using `examples/mongo-demo/src/contract.json` → blocked by pre-existing contract validation drift (F-3). Fell back to reading the mongo unit tests directly.

4. Reviewed `[Symbol.asyncDispose]` aliasing across all three facades:
   - Postgres: `[Symbol.asyncDispose](): Promise<void> { return this.close(); }` ✅
   - SQLite: same pattern ✅
   - Mongo: same pattern ✅

5. Confirmed the Postgres hang negative control: writing a script that creates `new Pool(...)` and exits WITHOUT calling `pool.end()` hangs for >2s (process stays alive). Writing the same script WITH `pool.end()` exits in ~1.5s. The fix is real and effective.

**Surprises / felt off:**

- The `pg.Pool` hang resolves in ~1.5s (not immediately). This is because `pool.end()` itself resolves quickly, but the process teardown drains the remaining libuv event-loop handles. Not a bug, but a developer might wonder if `db.close()` "worked" if they set a 1-second timeout. Consider noting in the skill that the exit may take up to 2s as libuv drains; 5s is a safe overall timeout for a clean exit test.

- The SQLite `ExperimentalWarning` from Node 24+ fires on every script using `@prisma-next/sqlite/runtime`. This is the first time it appears in the QA run; the skill doesn't mention it.
