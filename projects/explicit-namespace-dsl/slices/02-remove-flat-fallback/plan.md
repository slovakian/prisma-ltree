# Dispatch plan — slice 02: remove flat fallback + facade projection

Spec: [`./spec.md`](./spec.md). Parent plan: [`../../plans/plan.md`](../../plans/plan.md) (M1). Sequence is sequential per the single persistent implementer.

> **Grounding note (removal surface re-checked post-slice-01 merge).** The slice spec was written before slice 01's rebase landed and predicted that `TableNamesAcrossNamespaces` / `UnboundTables` "flat indexing is dead and is deleted." That prediction **no longer holds**: post-rebase, `UnboundTables<C>` and `TableNamesAcrossNamespaces<C>` were repurposed to mean *"the set of tables across all namespaces"* and are now load-bearing generic constraints for `TableProxy`, `TableProxyImpl`, and the whole `query-builder` package (`Ref`, `Root`, `SelectBuilder`, `selection.ts`). **The cut removes the flat *intersection member* of `Db<C>` / `OrmClient` and the flat runtime branches — it does NOT delete those helper types.** This is a plan-time grounding of the removal surface, which the spec explicitly deferred ("Dispatch plan deferred until slice 01 merges and the exact removal surface is grep-known"). The slice *outcome* (always-qualified builder; flat member gone; facade projects per `defaultNamespaceId`) is unchanged.

> **Current flat surface (what this slice removes):**
> - **SQL** (`packages/2-sql/4-lanes/sql-builder`): `Db<C>` is `{ [Name in TableNamesAcrossNamespaces<C>]: TableProxy } & { [Ns]: Namespace }`. `sql()`'s proxy `get()` falls back to `resolveTableForFlatName(storage, prop)` for non-namespace props. The flat intersection member + the flat fallback branch + the `resolveTableForFlatName` helper are the cut.
> - **ORM** (`packages/3-extensions/sql-orm-client`): `OrmClient` is `ModelCollectionMap<…> & NamespacedClientMap<…>`. `orm()`'s proxy `get()` falls back to `flatCollection(prop)` → `buildCollection(soleDomainNamespaceId(contract.domain), …)`. The flat member + `flatCollection` + the flat branch are the cut (`soleDomainNamespaceId` stays a foundation helper; only its use in `orm.ts` drops).
> - **Facades** (`postgres`, `sqlite`, `mongo`): expose `db.sql`/`db.orm` (mongo: `db.orm`/`db.query`, no `db.sql`) typed as the builder outputs directly. No projection helper exists yet.

## Dispatch 1: SQL builder flat cut

- **Outcome:** `Db<C>` is the namespaced-only mapped type (`{ [Ns in keyof storage.namespaces]: Namespace<C, Ns> }`); the flat-by-bare-name intersection member is gone. `sql()`'s proxy resolves namespace facets only — the `resolveTableForFlatName` fallback branch and the `resolveTableForFlatName` helper are removed. `resolveTableInNamespace` (the facet path) stays. `UnboundTables` / `TableNamesAcrossNamespaces` are **retained** (load-bearing for `TableProxy` / `query-builder`).
- **Builds on:** The spec's chosen design; the current `sql-builder` shape (additive intersection from slice 01).
- **Hands to:** `Db<C>` exposes namespace facets only; `sql.<ns>.<table>` is the sole builder shape; flat `sql.<table>` is a compile error and a runtime `undefined`. `sql-builder` package gate (typecheck + test + lint) is green; the workspace is expected to go red on facade type-tests + flat consumers (resolved D3/D4).
- **Focus:** `sql-builder` only — `src/types/db.ts` (drop the flat intersection member), `src/runtime/sql.ts` (drop the flat branch), `src/runtime/resolve-table.ts` (delete `resolveTableForFlatName`), `src/exports/types.ts` (drop the `Db` flat-member-only re-exports if any become unused). Update `test/types/namespaced-db.types.test-d.ts` + add a negative `@ts-expect-error` that flat `db.<table>` is gone. **Do not** touch `UnboundTables` / `TableNamesAcrossNamespaces` definitions or their proxy/query-builder consumers. No facade edits.
- **Tests-first / gates:** negative type test (flat `Db<C>` key absent) written before the cut; existing facet unit/type tests stay green. **F1 grep gate (package-scoped):** `rg 'resolveTableForFlatName' packages/2-sql/4-lanes/sql-builder` returns empty after the cut. Gate: `pnpm --filter @prisma-next/sql-builder typecheck` (+ test tsconfig) + `pnpm --filter @prisma-next/sql-builder lint` + `pnpm --filter @prisma-next/sql-builder test`.

## Dispatch 2: ORM client flat cut

- **Outcome:** `OrmClient` is the namespaced-only mapped type (`NamespacedClientMap` only); the flat `ModelCollectionMap` member is gone. `orm()`'s proxy resolves namespace facets only — the `flatCollection` path and the bare-model flat branch are removed (`orm.<ns>.<Model>` is the sole shape; bare `orm.<Model>` is a compile error and a runtime throw/`undefined` consistent with the facet path).
- **Builds on:** D1's shape (mirrors it in the ORM package; independent package, no shared write scope). `soleDomainNamespaceId` stays in foundation; its sole `orm.ts` consumer (`flatCollection`) is what goes.
- **Hands to:** `OrmClient` exposes namespace facets only; `orm.<ns>.<Model>` is the sole builder shape. `sql-orm-client` package gate green; workspace still red on facades/consumers (resolved D3/D4).
- **Focus:** `sql-orm-client` — `src/orm.ts` (drop `ModelCollectionMap` from `OrmClient`, delete `flatCollection` + `flatCache` + the bare-model branch in the proxy `get`). Update `test/orm.test.ts` (the "throws for unknown model name" path moves under the facet; retarget any flat-access assertions). **Do not** touch the namespace-facet path, `collection*.ts`, or foundation helpers. No facade edits.
- **Tests-first / gates:** negative type test (flat `OrmClient` key absent) first. **F1 grep gate:** `rg 'flatCollection|flatCache' packages/3-extensions/sql-orm-client/src` empty. Gate: `pnpm --filter @prisma-next/sql-orm-client typecheck` (+ test tsconfig) + lint + test.

> **Operator decision (between D2 and D3) — no shared helper; per-facade static projection.** The implementer's D3 R1 recon surfaced that the projection's `defaultNamespaceId` discriminator isn't reachable from any runtime descriptor (TML-2766). The operator ruled the whole helper/descriptor approach out as over-engineering: **each facade statically knows its own default namespace**, so it just returns the right shape directly. Postgres (multi-namespace) returns the qualified builder outputs (`db.sql = sql`, `db.orm = orm`). Sqlite and Mongo (single-namespace) return the unbound facet (`db.sql = sql.__unbound__`, `db.orm = orm.__unbound__`). No shared projection helper, no runtime dispatch, **no `framework-components` change** — fork (a) is closed by this decision. "No per-target switch" (AC5) is satisfied *a fortiori*: there is no central dispatcher to branch — each facade is statically its own target. AC5's type-level assertion becomes "each facade's exposed `sql`/`orm` type is the shape matching its target's default-namespace nature," asserted per facade. The original D3/D4 split (sql-family vs mongo) is retained only because mongo is structurally different (separate `mongo-orm`/`mongo-query-builder` packages, no `db.sql`).

## Dispatch 3: per-facade default-namespace projection — postgres + sqlite

No shared helper; each facade returns its shape directly. Both facade typechecks green.
- **Builds on:** D1 + D2 (builder outputs are namespaced-only; `sql.__unbound__` / `orm.__unbound__` are the namespace facets those cuts left in place).
- **Hands to:** postgres = qualified-only, sqlite = flat-via-`__unbound__`; `code-review.md` carries the AC4 (pg/sqlite) + AC5 (per-facade type-shape) evidence. D4 does the same for mongo; D5 migrates remaining flat consumers.
- **Focus:** `postgres/src/runtime/postgres.ts` + `sqlite/src/runtime/sqlite.ts` (the `sql`/`orm` construction sites, incl. transaction-context and postgres `prepare`) + their type-tests: `postgres/test/namespaced-facade.types.test-d.ts` (postgres → qualified-only, flat `db.sql.users`/`db.orm.User` become `@ts-expect-error`) + `sqlite/test/namespaced-facade.types.test-d.ts` (sqlite → flat `db.sql.users`/`db.orm.User` POSITIVE via `__unbound__`, qualified `db.sql.<ns>.<table>` becomes `@ts-expect-error`). **Reconcile the sqlite test fixture:** its sole namespace must be `__unbound__` (not `main`) so `sql.__unbound__` resolves — a fixture tweak; surface if it's more. **Mongo is D4 — do not touch it here.**
- **Note (FR11, not a blocker):** the projection is a static binding and adds no fail-fast logic. The ORM facet already throws a namespace-naming diagnostic (FR11 ✓ on the ORM side); the SQL facet returns `undefined` for an unknown table. Whether the SQL facet should also fail fast is a separate `sql-builder` follow-up (per `code-review.md § Orchestrator notes`), **not folded into D3**.
- **Tests-first / gates:** facade type-tests rewritten first (postgres qualified-only; sqlite flat-via-`__unbound__`; undeclared-ns negative). **AC5 evidence:** confirm there is no per-target `switch`/branch in either facade's projection (each just returns its shape) — `rg 'switch.*target|targetId ===|familyId ===|defaultNamespaceId' packages/3-extensions/{postgres,sqlite}/src/runtime` shows no projection-time dispatch; the per-facade type-tests assert the two shapes. ## Dispatch 4: mongo facade default-namespace projection — NO-OP (resolved)

- **Outcome (resolved — no code change):** D4 recon found mongo's ORM/query surfaces are **root-keyed, not namespace-keyed** — `MongoOrmClient` keys on `TContract['roots']` (`{ users, tasks }`); `db.query` is a method API (`{ from, rawCommand }`). Mongo was never part of the SQL-family namespaced-only cut and, being single-namespace, has no flat fallback to remove and no `__unbound__` facet to alias. The mongo facade is **already** in the intended unbound end-state: flat `db.orm.<Model>` works and is asserted green by `mongo/test/mongo.types.test-d.ts` (`keyof Db['orm']` = `'tasks' | 'users'`). **Orchestrator decision: close D4 as a no-op** (option (a)). Option (b) — converting `mongo-orm`/`mongo-query-builder` to namespace-keyed — is explicitly out of slice scope and unnecessary (no multi-namespace mongo contracts exist). See `learnings.md` § "Mongo's ORM/query surfaces are root-keyed".
- **Builds on:** D3 (confirmed the per-facade pattern; mongo simply needs none of it).
- **Hands to:** all three facades in end-state; AC4 fully covered (postgres qualified-only + sqlite flat-via-`__unbound__` + mongo already-flat-by-construction); AC5 per-facade evidence complete (mongo = flat root-keyed `db.orm`, no per-target switch). Spec AC4/AC5 mongo *wording* ("aliased to `orm.__unbound__`") describes a mechanism that doesn't apply — reconcile at close-out; the D6 ADR documents the root-keyed exception.
- **Focus:** no code change. Documentation only (this plan + `learnings.md` + the code-review scoreboard). The existing green mongo type-test is the AC4/AC5 mongo evidence.

## Dispatch 7: FR11 fail-fast on unknown namespace / table (spec-scoped, in-slice)

- **Outcome:** Accessing an **unknown namespace** on the builder surface (`sql.<unknownNs>` / `orm.<unknownNs>` from a runtime-widened contract) and an **unknown table in a known namespace** on the SQL facet (`sql.<ns>.<unknownTable>`) **fail fast with a diagnostic that names the namespace** (project FR11/FR2), instead of returning silent `undefined`. The ORM facet already throws "No model '<x>' in namespace '<ns>'" for unknown-model-in-namespace; this dispatch brings the SQL facet + the unknown-namespace cases to parity. This closes the FR11 gap the slice spec scoped to land "with the removal of the flat fallback path" but D1/D2 left as silent-`undefined`.
- **Builds on:** D1–D3 (the namespaced-only builder + facade projection are the surface FR11 guards).
- **Why in-slice (orchestrator intent-validation, D6 review):** the slice spec's pre-investigated edge case explicitly scoped "unknown namespace id from a runtime-widened contract → fail fast naming the namespace" to **this slice** ("Lands with the removal of the flat fallback path"). The ADR-proposed "defer to follow-up" drifts from that; FR11 is a slice-owned scoreboard AC. Landing it completes the slice, not expands it.
- **Focus:** `sql-builder/src/runtime/sql.ts` (proxy `get`: throw a namespace-naming diagnostic for a non-namespace top-level prop and for an unknown table within a namespace facet, instead of returning `undefined`) + the matching ORM top-level path in `sql-orm-client/src/orm.ts` if `orm.<unknownNs>` currently returns `undefined`. **Retarget the D1/D2 interim tests** that assert flat/unknown access → `undefined` to assert the fail-fast throw (a flat bare-table access IS an unknown-namespace access on the always-qualified builder — throwing names the namespace, which is strictly better than silent `undefined`). Keep the diagnostic message shape consistent with the ORM facet's existing "... in namespace '<ns>'"/available-list style.
- **Halt conditions:** retargeting reveals a consumer (in `packages/`/`examples/`/tests) that legitimately relies on flat access returning `undefined` rather than throwing → surface (that would be a consumer the migration missed). A workspace gate goes red from the new throw beyond the expected interim-test retargets → surface.
- **Tests-first / gates:** test (FIRST) asserting `sql.<unknownNs>` and `sql.<ns>.<unknownTable>` throw a namespace-naming diagnostic; ORM parity test; single-namespace happy-path unchanged. Gate: `pnpm --filter @prisma-next/sql-builder --filter @prisma-next/sql-orm-client typecheck` (+ test tsconfig) + lint + test; then `pnpm build` + workspace `pnpm typecheck` + `pnpm test:packages` green (the retargets must not break the migrated consumers — they qualified to *valid* namespaces, so they won't hit the throw); `pnpm lint:deps`.
